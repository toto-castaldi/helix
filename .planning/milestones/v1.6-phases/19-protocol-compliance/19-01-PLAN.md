Three changes to the main Deno.serve() handler and handleJsonRpc() function:

**Change 1 — Notification detection at HTTP level (PROTO-02):**
After the JSON body is parsed (line ~2443 `const body = JSON.parse(bodyText) as JsonRpcRequest`) and BEFORE the `if (body.method === "initialize")` check (line ~2448), add notification detection:

```typescript
// Detect JSON-RPC notification (no "id" field) — MCP 2025-03-26 Streamable HTTP
// Spec: "If the input consists solely of JSON-RPC notifications,
// the server MUST return HTTP status code 202 Accepted with no body."
const isNotification = !Array.isArray(body) && typeof body === "object" && body !== null && !("id" in body)
const isBatchNotification = Array.isArray(body) && body.length > 0 && body.every((msg: unknown) =>
  typeof msg === "object" && msg !== null && !("id" in (msg as Record<string, unknown>))
)

if (isNotification || isBatchNotification) {
  console.log("[RPC] Notification received:", isNotification ? (body as Record<string, unknown>).method : "batch")
  return new Response(null, {
    status: 202,
    headers: corsHeaders,
  })
}
```

This must appear BEFORE the authentication check because `notifications/initialized` is sent immediately after `initialize` (which itself skips auth) and may not include the API key.

**Change 2 — Protocol version string update (PROTO-01):**
Update the `protocolVersion` string in BOTH locations:

1. Inside `handleJsonRpc()` function, in the `case "initialize":` block (line ~2284):
   Change `protocolVersion: "2024-11-05"` to `protocolVersion: "2025-03-26"`

2. In the early-return initialize block inside `Deno.serve()` (line ~2454):
   Change `protocolVersion: "2024-11-05"` to `protocolVersion: "2025-03-26"`

Search the ENTIRE file for the string `"2024-11-05"` and ensure ALL occurrences are replaced. There should be exactly 2.

**Change 3 — Update initialized case in handleJsonRpc (defense in depth):**
In `handleJsonRpc()`, change `case "initialized":` (line ~2290) to `case "notifications/initialized":` to match the 2025-03-26 spec method name. With notification detection at HTTP level, this case should never be reached, but it provides defense in depth if a notification somehow passes the HTTP-level check.

**Do NOT change the GET 405 handling** (lines 2417-2430). The current implementation already returns HTTP 405 with `Allow: POST, OPTIONS` header and CORS headers, which is spec-compliant (PROTO-03). The JSON error body is informative and does not violate the spec.
