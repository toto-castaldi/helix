Create `scripts/test-mcp-e2e.sh` as an executable bash script that tests the full MCP server.

**IMPORTANT: The script assumes local Supabase is already running** (`npm run supabase:start`) and Edge Functions are being served (`npx supabase functions serve --env-file supabase/.env`). The script does NOT start these services -- it documents them as prerequisites in a usage comment at the top.

**Script structure:**

1. **Header and helpers:**
   - `#!/bin/bash` with `set -euo pipefail`
   - Usage comment block explaining prerequisites (Supabase running, Edge Functions served)
   - `MCP_URL="http://127.0.0.1:54321/functions/v1/helix-mcp"` constant
   - Color codes for pass (green), fail (red), info (blue)
   - Counter variables: `PASS=0`, `FAIL=0`, `TOTAL=0`
   - Helper function `mcp_call(method, params, id)` that wraps curl: `curl -s -X POST $MCP_URL -H "Content-Type: application/json" -H "X-Helix-API-Key: $TEST_API_KEY" -d '{"jsonrpc":"2.0","method":"'"$1"'","id":'"$3"',"params":'"$2"'}'`
   - Helper function `mcp_call_no_auth(method, params, id)` same but without API key header
   - Helper function `assert_result(test_name, response)` that checks: (a) response is valid JSON, (b) has `"result"` key, (c) does NOT have `"error"` key. Increments PASS or FAIL.
   - Helper function `assert_error(test_name, response)` that checks response has `"error"` key. Increments PASS or FAIL.
   - Helper function `assert_tool_result(test_name, response)` that checks: (a) valid JSON, (b) has `result.content` array, (c) does NOT have `result.isError`. Increments PASS or FAIL.
   - Helper function `assert_tool_error(test_name, response)` that checks `result.isError` is true. Increments PASS or FAIL.
   - Helper function `assert_http_status(test_name, expected_status, actual_status)` for HTTP status checks.

2. **Setup phase:**
   - Generate test API key: `TEST_API_KEY="hx_test_$(openssl rand -hex 16)"`
   - Compute SHA-256 hash: `HASH=$(echo -n "$TEST_API_KEY" | sha256sum | cut -d' ' -f1)`
   - Create test user in auth.users via SQL: `PGPASSWORD=postgres psql -h 127.0.0.1 -p 54322 -U postgres -d postgres -c` with an INSERT into `auth.users` (id = fixed UUID `'aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee'`, email = `test@helix.local`). Use `ON CONFLICT DO NOTHING` in case it already exists.
   - Insert API key hash: `INSERT INTO public.coach_ai_settings (user_id, helix_mcp_api_key_hash) VALUES ('aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee', '$HASH') ON CONFLICT (user_id) DO UPDATE SET helix_mcp_api_key_hash = '$HASH';`
   - Create test client: INSERT INTO public.clients with user_id, first_name='Test', last_name='Client', gender='male'. Capture CLIENT_ID.
   - Create test gym: INSERT INTO public.gyms with user_id, name='Test Gym', address='Via Test 1'. Capture GYM_ID.
   - Print "Setup complete" with API key info (masked).

3. **Protocol tests (4 tests):**
   - `initialize`: Send method `initialize` with params `{"protocolVersion":"2025-03-26","capabilities":{},"clientInfo":{"name":"test","version":"1.0"}}` WITHOUT auth header. Assert result contains `protocolVersion`.
   - `notifications/initialized`: Send notification (no `id` field!) via curl checking HTTP status code. Use `curl -s -o /dev/null -w "%{http_code}"`. Assert HTTP 202.
   - `GET 405`: Send GET request to MCP_URL, assert HTTP 405.
   - `ping`: Send method `ping` WITH auth. Assert result exists (empty object).

4. **Auth tests (2 tests):**
   - No API key: Call `resources/list` without auth header. Assert error response.
   - Invalid API key: Call `resources/list` with `X-Helix-API-Key: hx_invalid_key`. Assert error response.

5. **Resource list test (1 test):**
   - Call `resources/list`. Assert result. Extract and print count of resources (should be 20).

6. **Concrete resource read tests (9 tests):**
   - Read each concrete URI: `helix://clients`, `helix://gyms`, `helix://exercises`, `helix://exercises/tags`, `helix://sessions`, `helix://sessions/planned`, `helix://group-templates`, `helix://coach/summary`, `helix://today`. Assert result for each.

7. **Tool tests (via tool calls that create test data -- IDs captured for later):**
   - `create_session` with CLIENT_ID and session_date "2026-03-01" and gym_id GYM_ID. Capture SESSION_ID from response using jq.
   - `add_session_exercise` with SESSION_ID and a default exercise ID (query one from DB or use the first from seed data -- capture EXERCISE_ID first via SQL: `SELECT id FROM public.exercises WHERE user_id IS NULL LIMIT 1`). Capture SESSION_EXERCISE_ID.
   - `update_session` with SESSION_ID, notes "Test notes".
   - `update_session_exercise` with SESSION_EXERCISE_ID, sets 3, reps 12.
   - `duplicate_session` with SESSION_ID, new_date "2026-03-02". Capture DUPLICATE_SESSION_ID.
   - `complete_session` with DUPLICATE_SESSION_ID.
   - `reorder_session_exercises` with SESSION_ID and exercise_ids array containing SESSION_EXERCISE_ID.
   - `create_training_plan` with CLIENT_ID, session_date "2026-03-03", exercises array with one exercise (use exercise name from seed data, e.g. "Squat" or query the name). Capture PLAN_SESSION_ID.
   - `create_group_template` with name "Test Template". Capture TEMPLATE_ID.
   - `update_group_template` with TEMPLATE_ID, name "Updated Template".
   - `add_template_exercise` with TEMPLATE_ID and EXERCISE_ID. Capture TEMPLATE_EXERCISE_ID.
   - `apply_template_to_session` with TEMPLATE_ID, SESSION_ID, mode "append".

   Assert tool result (not error) for each call above.

8. **Templated resource read tests (11 tests)** using captured IDs:
   - `helix://clients/{CLIENT_ID}`
   - `helix://clients/{CLIENT_ID}/card`
   - `helix://clients/{CLIENT_ID}/goals`
   - `helix://clients/{CLIENT_ID}/sessions`
   - `helix://gyms/{GYM_ID}`
   - `helix://exercises/{EXERCISE_ID}`
   - `helix://exercises/{EXERCISE_ID}/lumio`
   - `helix://exercises/tags/{TAG}` (query a tag from DB first)
   - `helix://sessions/date/2026-03-01`
   - `helix://sessions/{SESSION_ID}`
   - `helix://group-templates/{TEMPLATE_ID}`

   Assert result for each.

9. **Tool list test (1 test):**
   - Call `tools/list`. Assert result. Print count (should be 16).

10. **Prompt tests (6 tests):**
    - `prompts/list`: Assert result. Print count (should be 5).
    - `prompts/get` for each prompt: `plan-session` (with args client_id=CLIENT_ID), `weekly-plan` (with args client_id=CLIENT_ID, start_date="2026-03-01", sessions_count="3"), `session-review` (with args session_id=SESSION_ID), `daily-briefing` (no args), `template-analysis` (no args). Assert result for each.

11. **Validation tests (3 tests):**
    - `create_session` with `client_id: "not-a-uuid"` -- assert tool error with validation message.
    - `create_session` with missing `session_date` -- assert tool error.
    - `update_session` with missing `session_id` -- assert tool error.

12. **Destructive tool tests (4 tests -- run last):**
    - `remove_template_exercise` with TEMPLATE_EXERCISE_ID.
    - `delete_group_template` with TEMPLATE_ID.
    - `remove_session_exercise` with SESSION_EXERCISE_ID.
    - `delete_session` with SESSION_ID.
    Assert tool result for each.

13. **Cleanup:**
    - Delete test data via SQL: DELETE from sessions, clients, gyms, coach_ai_settings for the test user. DELETE auth.users for test user.
    - Print summary: `echo "Results: $PASS passed, $FAIL failed out of $TOTAL tests"`
    - Exit with code 0 if FAIL==0, else exit 1.

**Important notes:**
- Use `jq -r` to extract values from JSON responses (session IDs, exercise IDs, etc.).
- For capturing IDs from tool results, the pattern is: `echo "$RESPONSE" | jq -r '.result.content[0].text' | jq -r '.id'` (tool results have content[0].text as a JSON string).
- Increment ID counter for each JSON-RPC request to keep them unique.
- Use `set +e` around test sections (re-enable with `set -e` after) to prevent script from exiting on first test failure -- accumulate pass/fail counts.
- Make the script executable: `chmod +x scripts/test-mcp-e2e.sh`.
