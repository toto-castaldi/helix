## Version Extraction Verification

The `Extract milestone version` step uses two-tier fallback logic:

**Tier 1 (PRIMARY):** `grep -oP 'Current Milestone: \Kv\d+\.\d+' .planning/PROJECT.md`
- Tested against actual file: returns `v1.5` (PROJECT.md line 45 contains `**Current Milestone: v1.5 Versioning GSD**`)
- Status: WORKS

**Tier 2 (FALLBACK):** `grep -oP '## \Kv\d+\.\d+' .planning/MILESTONES.md | sort -rV | head -1`
- Tested against actual file: returns `v1.4` (highest version in shipped milestones)
- Status: WORKS — correctly yields the most recent shipped milestone if PROJECT.md primary fails

**Tier 3:** `"dev"` — if neither file yields a version
- Status: CORRECT safe default

**Note:** The fallback produces `v1.4` (not `v1.5`) because MILESTONES.md only records shipped milestones; v1.5 is active but not yet shipped. This is intentional and correct: the primary path reads PROJECT.md which holds the active milestone. The fallback would only activate if PROJECT.md's "Current Milestone:" line is absent or malformatted, in which case reverting to the last shipped version is the appropriate safe behavior.

## Additional Observations

### TypeScript type coverage
`src/vite-env.d.ts` line 7 declares `readonly VITE_APP_VERSION: string` in `ImportMetaEnv`. All three consumers (`Layout.tsx`, `TabletDateSelect.tsx`, `main.ts`) will get correct TypeScript types when accessing `import.meta.env.VITE_APP_VERSION`.

### Dev-mode fallback
All three consumers use `import.meta.env.VITE_APP_VERSION || 'dev'`. When running locally without the env var set, the displayed value is `'dev'` — a clear indicator that the build-time injection has not occurred. This is correct behavior.

### Landing build does not use TypeScript compilation
`build:landing` runs `vite build --config vite.config.landing.ts` without `tsc -b` (unlike `build` and `build:live` which run `tsc -b && vite build`). The landing entry is `src/landing/main.ts` (vanilla TS, no React). This is intentional and consistent with the vanilla JS approach; Vite handles the TS transpilation directly. No integration issue here.

### DEPLOY_PATH_LANDING secret
`deploy.yml` line 84 references `secrets.DEPLOY_PATH_LANDING`. This secret is not listed in the CLAUDE.md GitHub Secrets table (which only lists `DEPLOY_PATH` and `DEPLOY_PATH_LIVE`). This is a pre-existing documentation gap from v1.4, not introduced by v1.5. It does not affect v1.5 integration correctness but should be noted for CLAUDE.md upkeep.
